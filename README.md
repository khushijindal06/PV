# ML Pipeline — Pemphigus Vulgaris Classifier

Everything for the PV classification model — data, code, and generated
outputs — lives in this one folder.

## Contents

```
ml_pipeline/
├── data/
│   └── pemphigus_dataset_dummy_300.csv   # the real (dummy) project dataset, 300 patients
├── src/
│   ├── common.py            # shared feature lists + fixed train/test split
│   ├── train_baseline.py    # original pipeline from the Progress Report (RF + GridSearchCV)
│   ├── train_improved.py    # updated pipeline (SMOTE, model comparison, PV screening, SHAP)
│   └── evaluate_compare.py  # prints/saves the before/after comparison
├── artifacts/                # generated on each run (metrics, plots, models)
│   ├── baseline_metrics.json / baseline_confusion_matrix.png
│   ├── improved_metrics.json / improved_confusion_matrix.png
│   ├── shap_summary_pv.png
│   └── comparison_report.md
└── requirements.txt
```

`artifacts/*.joblib` (the trained model files) are gitignored — they're
regenerated by re-running the scripts below, so only the human-readable
metrics/plots/reports are committed.

## The dataset

`data/pemphigus_dataset_dummy_300.csv` is the real file provided for this
project: 300 patients, 4-class diagnosis target (`Healthy`, `Pemphigus
Vulgaris`, `Pemphigus Foliaceus`, `Bullous Pemphigoid`), with the clinical
schema from the Progress Report (age, gender, ethnicity, initial symptom
location, Nikolsky sign, anti-Dsg3/Dsg1 ELISA titres, DIF result).

This is a genuine 4-class problem, not a binary PV-vs-control one — which
actually matches the Progress Report's stated objective more closely
(section 2.2/3.1: distinguish PV from *both* healthy controls and other
similar blistering disorders like Bullous Pemphigoid and Pemphigus
Foliaceus).

**Important finding:** exploratory analysis shows this dataset carries no
real diagnostic signal — per-class means and standard deviations for age,
Nikolsky sign, and both ELISA titres are statistically indistinguishable
across all four diagnosis groups. Both the baseline and improved models
score at or below chance level (~20% and ~13% accuracy respectively,
against a 25% random baseline for four balanced classes; ROC-AUC ~0.43-0.45
against 0.5 for random). This is expected and consistent with the Progress
Report itself, which calls this a placeholder/dummy dataset (section 5,
section 9: "all performance figures are illustrative and must be
revalidated on real clinical data"). No amount of modelling can manufacture
signal that isn't in the data — see "Result summary" below for what the
pipeline upgrades demonstrably do instead.

## Running

```bash
python -m venv .venv && source .venv/bin/activate
pip install -r ml_pipeline/requirements.txt

# from the repository root:
python -m ml_pipeline.src.train_baseline    # writes artifacts/baseline_*
python -m ml_pipeline.src.train_improved    # writes artifacts/improved_*
python -m ml_pipeline.src.evaluate_compare  # writes artifacts/comparison_report.md
```

## What "improved" means here

Both models are trained/evaluated on an identical stratified 80/20 split of
the same dataset. `train_improved.py` adds, on top of the report's original
Random Forest + GridSearchCV pipeline:

- **Missing-value indicator columns** on the numeric features (the report
  calls missingness "potentially informative")
- **SMOTE** oversampling inside the CV loop for class imbalance (report:
  "evaluated for future use")
- **A comparison across model families** — Random Forest, Extra Trees,
  Gradient Boosting, Logistic Regression, plus a soft-voting ensemble of
  the best two — instead of committing to Random Forest alone, with the
  number of selected features (`k`) also tuned as a hyperparameter
- **A PV-vs-rest sensitivity screen**: since the report's clinical concern
  is specifically about *missing PV cases* (not misclassifying among the
  other three diseases equally), a separate decision threshold is tuned on
  P(Pemphigus Vulgaris) to flag high-suspicion PV cases that the plain
  multi-class argmax decision would otherwise miss
- **SHAP feature importance** for the Pemphigus Vulgaris class (report
  lists XAI integration as "Planned")

On this dataset, none of that can lift accuracy above chance, because the
data has no signal to find (see the finding above) — the PV-screening
threshold actually collapsing to a near-zero value that flags almost every
patient as PV is itself a symptom of that: the optimizer found no
non-trivial threshold worth preferring. What this pipeline demonstrably
delivers is: (a) an honest, reproducible before/after comparison rather
than cherry-picked numbers, and (b) a complete, ready-to-run pipeline
(preprocessing, imbalance handling, model selection, threshold tuning,
explainability) that will produce meaningful results the moment it's
pointed at a dataset with genuine clinical signal — real patient data, or
even a dummy dataset deliberately constructed with the report's described
correlations (higher Dsg3/Dsg1 titres, positive DIF, Nikolsky sign in PV
cases, etc.).

Full numbers are in `artifacts/comparison_report.md` after running the
scripts above.
